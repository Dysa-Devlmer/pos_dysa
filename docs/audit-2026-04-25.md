## 4 · DB / Migration Audit (skill: db-migration)

**Schema:** 13 modelos · 3 enums · `Venta→DetalleVenta` Cascade · `Account/Session→Usuario` Cascade.

### 🔴 Riesgos data layer

- **Sin `prisma/migrations/`** confirmado (gotcha G-M32). `db push` no genera historial → **rollback en prod = imposible sin pg_dump**.
- **Sin AuditLog.** Ley 21.719 (vigente Dic 2026) exige bitácora de accesos a `Cliente.rut/email/direccion`. Riesgo no-cumplimiento.
- **Sin backup offsite.** Cron diario existe (`/etc/cron.daily/pos-backup`, retention 30d, gzip) pero **mismo disco que la BD** + sin alerta de fallo.
- **Soft-delete inconsistente.** `Producto.activo`, `Categoria.activa`, `Usuario.activo` (soft) vs `Venta`/`DetalleVenta`/`Devolucion` hard-delete con Cascade. **Riesgo SII**: boletas eliminadas físicamente violan retención tributaria (6 años CL).

### 🟠 Performance — índices faltantes

- `Venta.metodoPago` (filtros dashboard).
- `Producto.activo + stock` (alertas stock bajo escanean tabla).
- `Cliente.compras` / `ultimaCompra` (top clientes).
- `Devolucion.esTotal`.
- N+1 probable en server actions de Venta sin `include`.

### Plan migration formal (2-3h)

1. `pnpm --filter @repo/db prisma migrate dev --name init --create-only` → genera SQL del estado actual.
2. `prisma migrate resolve --applied init` en local + prod → marca baseline sin re-correr DDL.
3. Reemplazar `db:push` por `db:migrate deploy` en `scripts/deploy.sh`.
4. Actualizar gotcha G-M32 + `memory/context/stack-tech.md`.

**Riesgo:** medio (baseline mal marcada → re-aplica DDL → FK violations). Mitigación: snapshot Vultr.

## 5 · Infra & Deploy Audit (skill: docker-deploy)

**Stack:** `node:22-alpine` multi-stage 3-etapas, Next.js standalone, healthcheck `wget /api/health` (DB connectivity), volumes `pos-postgres-data`/`pos-pgadmin-data`, network `pos-chile-network` bridge.

### 🔴 Riesgos críticos

- **Secrets en plaintext.** `.env.docker` por scp al VPS → `/opt/pos-chile/.env.docker` legible por root. Sin Docker secrets / Vault.
- **Backup BD no offsite.** Mismo disco. Sin alerta de fallo de `pg_dump`.
- **Rollback no cubre BD.** Solo restaura archivos. Migración Prisma destructiva = sin rollback.
- **SPOF.** 1 VPS Vultr · 1 Postgres · 0 Redis local (Upstash gestionado OK).
- **Logs no agregados.** Sin `logging:` driver, sin ELK/Loki. Sentry captura solo errores aplicativos.

### 🟠 Mejoras prioritarias

- **CI/CD:** `has_ci=false`. Recomendado GitHub Actions: `pnpm type-check + test + build` pre-merge a `main`.
- **Build en VPS.** Mover a CI → push imagen a GHCR → VPS solo `pull + up`.
- **deploy.sh hueco real:** si falla en fase 4 (rsync) o fase 5 (build), el web stop ya ocurrió y `set -e` aborta sin restaurar.

### 🟡 Hardening

- Imágenes pinned por minor (`postgres:16-alpine`) pero no por SHA. `dpage/pgadmin4:latest` drift no controlado.
- Sin `mem_limit`/`cpus`. Memory leak puede tumbar VPS entero.
- Sin `read_only: true` + `tmpfs` (trivial dado que avatares son base64 en DB).
- Sin `HEALTHCHECK` directive en Dockerfile (solo en compose, aceptable).

> Complementa la sección 1 (`/code-reviewer` que detectó bugs específicos). Esta revisa **patterns sistémicos**.

### 🟢 Lo que está bien implementado (top 5)

1. **PrismaClient singleton via `globalThis`** (`packages/db/src/client.ts:11-29`) — evita múltiples instancias en hot-reload y serverless re-eval.
2. **Lock pesimista `SELECT ... FOR UPDATE NOWAIT`** en `crearDevolucion` (`devoluciones/actions.ts:69-71`, `api/v1/devoluciones/route.ts:129-131`) — serializa concurrencia sin deadlocks.
3. **Validación Zod ANTES de tocar BD** en todas las server actions — datos malformados nunca llegan a Prisma.
4. **`Promise.all([findMany, count])`** en endpoints paginados — queries paralelas, no secuenciales.
5. **Pre-checks defensivos antes de eliminar** (count FK + catch P2003 fallback) — UX consistente.

### 🔴 Anti-patterns críticos (3)

**B1. N+1 en stock updates dentro de transactions críticas.**
Archivos: `ventas/actions.ts:185-193, 257-265, 421-430, 466-474` · `api/v1/ventas/route.ts:131-139` · `api/v1/ventas/[id]/route.ts:46-54` · `devoluciones/actions.ts:178-186`.
Cada item del carrito dispara un `tx.producto.update()` separado → carrito de 10 items = 10 round-trips dentro del lock.
**Costo:** transaction holds locks N× más tiempo, agrava contención bajo carga.
**Fix:** `Promise.all(items.map(...))` para paralelizar dentro de la tx (mínimo), o `$executeRaw UPDATE ... WHERE id IN (...)` agrupado.

**B2. Stock check no atómico (TOCTOU).**
`ventas/actions.ts:118-132` y `api/v1/ventas/route.ts:90-98` — el `findMany` que valida stock está **fuera** de la `$transaction`. Entre validación y `decrement`, otra venta puede dejar `stock < 0`. Schema sin CHECK constraint `stock >= 0`.
**Costo:** stock negativo silencioso bajo concurrencia.
**Fix:** mover dentro del `$transaction` con `SELECT ... FOR UPDATE`, o `UPDATE producto SET stock = stock - $cantidad WHERE id = $id AND stock >= $cantidad RETURNING stock` y abortar si `rowCount === 0`. Agregar CHECK constraint a nivel BD.

**B3. Sin `isolationLevel` ni `timeout` explícito en transactions largas.**
Default Prisma: `ReadCommitted` con timeout 5s. `crearDevolucion` (8+ pasos) y `editarVenta` (revierte+aplica) corren riesgo de phantom reads y timeouts en cargas pesadas.
**Fix:** `prisma.$transaction(fn, { isolationLevel: 'Serializable', timeout: 15000, maxWait: 5000 })` en devoluciones.

### 🟠 Mejoras prioritarias (top 5)

1. **Sin `select` explícito en `findUnique`/`findFirst`** — ~40% over-fetch en chequeos de existencia. Impacto: -30% bytes red.
2. **Sin audit trail (`updatedBy`)** — `editarVenta` sobrescribe `usuarioId` (borra creador original). `actualizarProducto/Cliente/Categoria` no registran editor. **Bloqueante SII regulatorio.**
3. **`include: { detalles: true }` sin `select` anidado** — trae todos los campos cuando solo se usan `productoId, cantidad`.
4. **Manejo genérico errores Prisma** — solo `eliminarVenta` distingue P2003. Resto cae al catch genérico exponiendo "Unique constraint failed on (codigoBarras)" al user.
5. **API REST diverge de Server Action** — `numeroBoleta` distinto, IVA hardcoded inline, sin descuentos. **Fix:** extraer dominio a `lib/domain/venta.ts` y consumir desde ambos endpoints (single source of truth).
